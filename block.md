# Tiny-LSM Block 模块原理解析

在 LSM-Tree 架构中，SSTable 通常由多个数据块（Block）组成，Block 也是磁盘读取和缓存在内存中的最小单位（Block Cache）。在 `Tiny-LSM` 中，`Block` 扮演着非常底层的存储单元角色。

以下是从源码级别对 Block 的编码解码、构建、二分查询以及范围查询的原理拆解。

## 1. Block 的内存布局与编解码 (Encode / Decode)

为了能够高效地落盘并支持快速检索，Block 在内存和磁盘中的布局需要十分紧凑。

**整体块布局**：
```text
--------------------------------------------------------------------------------------
|             Data Section           |      Offset Section |          Extra          |
--------------------------------------------------------------------------------------
|Entry#1|Entry#2|...|Entry#N         |Offset#1|...|Offset#N| num_elements (2B) + Hash|
--------------------------------------------------------------------------------------
```

**单条记录 (Entry) 的布局**：
```text
---------------------------------------------------------------------
|key_len(2B)| key(keylen) |val_len(2B)| val(vallen) |tranc_id(8B)   | 
---------------------------------------------------------------------
```

### 编码机制 (Encode)
编码（落盘前）的过程非常简单粗暴，就是一次纯粹的内存拼接：
1. `Block` 内部维护了两个动态数组：`std::vector<uint8_t> data` 存放所有 Entry，`std::vector<uint16_t> offsets` 存放每个 Entry 在 `data` 中的起始偏移量。
2. 编码时，按顺序拷贝 `data` 段的字节流。
3. 紧接着拷贝 `offsets` 数组的字节流。
4. 在末尾追加总元素个数（`num_elements`，2字节），以及可选的整个块的 Hash 校验和（4字节）。

### 解码机制 (Decode)
解码（从磁盘读入内存后）的过程采用**“倒序推理”**：
1. 因为 `data` 段的长度不固定，所以程序先读取末尾的 Hash 校验数据完整性，再读取倒数前 2 个字节得到 `num_elements`。
2. 既然知道偏移量都是 2 字节（`uint16_t`），就可以推算出整个 `Offset Section` 占用的字节数（`num_elements * 2`）。
3. 知道 `Offset Section` 的大小和起始位置后，前面剩下的所有字节自然就是 `Data Section`。
4. 将这两段内容分别反序列化进 `data` 数组和 `offsets` 数组，Block 就瞬间在内存中重建完毕了。

## 2. Block 的构建机制 (Build)

构建 Block 是通过不断调用 `add_entry` 函数完成的，本质上是一个 **Append-only（只追加）** 的过程：

1. **容量预检**：在追加前，会预判新加入的 Entry 加上偏移量占用的空间是否会超出 `Block` 预设的 `capacity`（通常是 4KB）。如果超出且允许拒绝，则返回 `false` 提示上层去开辟新的 Block。
2. **追加数据**：将传入的 `key`、`value` 和 `tranc_id` 按照上述的 Entry 格式组装，追加到 `data` 数组的末尾。
3. **记录偏移量**：追加前记录下当前的 `data.size()`，这个值就是新 Entry 的起始偏移量。将这个偏移量推入 `offsets` 数组的末尾。
4. **有序保证**：需要注意的是，`add_entry` 本身不负责排序。因为数据是先经过 SkipList 排序后按序刷盘的，所以上层（SSTable Builder）会保证**传入的 Key 始终是字典序递增的，相同的 Key 则是按 `tranc_id` 递减的**。

## 3. 二分查询 (Binary Search)

由于 `offsets` 数组对应的数据在物理上是有序的，Block 原生支持高效的 `O(log N)` 二分查找。实现逻辑在 `get_idx_binary` 函数中：

1. **基于 Offsets 的二分**：指针 `left` 和 `right` 并不在 `data` 上跳跃，而是在 `offsets` 数组的下标区间 `[0, num_elements - 1]` 上做二分。
2. **解析比较**：取 `mid` 下标对应的偏移量 `offsets[mid]`，去 `data` 中精确解析出对应位置的 Key，与目标 Key 进行比较 (`compare_key_at`)。如果大了去左半边，小了去右半边。
3. **MVCC 可见性回溯 (`adjust_idx_by_tranc_id`)**：
   * 这是 Tiny-LSM 的精髓之一。如果在二分时碰巧命中了一个版本，这仅仅是找到了这个 Key 的“某个”历史版本。
   * 因为相同的 Key 连续存放在一起，且按 `tranc_id` 从大到小排列，程序会从 `mid` 开始**向前后探测（游走）**，直到找到**第一个 `tranc_id` 小于等于当前读事务 ID 的有效版本**，并返回它最终在 `offsets` 中的下标索引。

## 4. 迭代器与范围查询 (Iterator & Range Query)

单点查询（Point Get）只能拿一条数据，为了支持 LSM 引擎级别的全表扫描，Block 还必须支持迭代与区间提取。

### 块迭代器 (`BlockIterator`)
非常轻量级，内部只持有三个信息：
* 指向所属 Block 的共享指针 `std::shared_ptr<Block>`。
* 当前正在访问的 `offsets` 下标索引 `idx`。
* 当前读事务的 `tranc_id`。

在调用 `operator++` 推进时，实质上就是 `idx++`，然后通过偏移量去解析下一条数据。迭代器内部会自动封装逻辑，跳过那些不可见的未来事务数据和删除标记。

### 范围查询 (`get_monotony_predicate_iters`)
和 SkipList 类似，Block 的范围查询同样抽象为了**基于谓词的区间搜索**。它利用块内部数据的严格有序性，使用**两次二分查找**来闪电般地切出目标区间：

1. **寻找左边界（第一次二分）**：使用二分查找在 `offsets` 中寻找到**第一个满足谓词条件**（`predicate == 0`）的数据下标位置，记为 `first`。
2. **寻找右边界（第二次二分）**：依然是二分查找，但是这次向右施压，寻找**最后一个满足谓词条件**的数据下标位置，记为 `last`。
3. **包装区间**：拿到区间 `[first, last]` 后，将其包装为 `BlockIterator` 返回。为了符合 C++ 迭代器的开区间标准，右边界会用 `last + 1` 包装，最终向上层返回一个 `[begin, end)` 的有效迭代器对。

这种设计使得在即使拥有上千条记录的 Block 中进行前缀查询或范围 Scan 时，不需要进行任何的线性扫描试错，两次二分（`O(log N)`）即可精准圈定有效数据范围。
